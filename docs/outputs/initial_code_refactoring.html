2 — AdminOptim.java ARCHITECTURE

A1 — Replace double[N][N] with Set<Integer>[] adjacency lists

Location: AdminOptim.java — class fields + constructor

Issue type: Structure / Performance

// BEFORE
private double[][] W;
private int[] followerNumArray;
public AdminOptim(int userNum, double[][] W) {
    this.n = userNum;
    this.W = W;
    this.followerNumArray = new int[n];
}

// AFTER
private Set<Integer>[] followees;  // agents i follows
private Set<Integer>[] followers;  // agents that follow j
private int[] followerNumArray;    // cached in-degree; updated incrementally
@SuppressWarnings("unchecked")
public AdminOptim(int userNum, double[][] W) {
    this.followees = new HashSet[n];
    this.followers = new HashSet[n];
    for (int i = 0; i < n; i++) { followees[i] = new HashSet<>(); followers[i] = new HashSet<>(); }
    for (int i = 0; i < n; i++)
        for (int j = 0; j < n; j++)
            if (W[i][j] > 0.0) { followees[i].add(j); followers[j].add(i); followerNumArray[j]++; }
}

Rationale: With MAX_FOLLOW_CAPACITY = 10, edge density is ≤1%. The old dense matrix allocated 8 MB (1000×1000 doubles) and required O(N) column/row scans for every follower lookup. The adjacency lists reduce memory ~100× and graph traversal from O(N) to O(degree ≈ 10). The one-time O(N²) constructor cost is paid only at initialization.

A2 — getAdjacencyMatrix() becomes an on-demand generator

Location: AdminOptim.java:getAdjacencyMatrix()

Issue type: Structure

// BEFORE: returned a deep copy of the stored matrix
public double[][] getAdjacencyMatrix() {
    double[][] copy = new double[n][n];
    for (int i = 0; i < n; i++) copy[i] = Arrays.copyOf(this.W[i], n);
    return copy;
}

// AFTER: reconstructs from adjacency sets on demand
public double[][] getAdjacencyMatrix() {
    double[][] matrix = new double[n][n];
    for (int i = 0; i < n; i++)
        for (int j : followees[i])
            matrix[i][j] = 1.0;
    return matrix;
}

Rationale: The method is now only called at 5000-step checkpoints (8 times per 40 000-step run) and at simulation init. Paying O(N²) rebuild 8 times is acceptable; paying it every step is not.

A3 — New getFollowers() and getFollowees() accessors

Location: AdminOptim.java — new methods

Issue type: Structure

public Set<Integer> getFollowers(int userId) { return followers[userId]; }
public Set<Integer> getFollowees(int userId) { return followees[userId]; }

Rationale: Returns the live set by reference (O(1)). Callers iterate it for post/repost distribution — O(degree) instead of O(N) scan. Callers must not mutate the set during iteration; in the current code only updateAdjacencyMatrix() is a writer and it is called after the per-agent inner block.

A4 — updateAdjacencyMatrix() becomes O(1) incremental

Location: AdminOptim.java:updateAdjacencyMatrix()

Issue type: Performance

// BEFORE: after any follow/unfollow, full O(N²) rescan
public void updateAdjacencyMatrix(int userId, int[] followedIds, int unfollowedId) {
    if (followedIds[0] >= 0) this.W[userId][followedIds[0]] = 1.0;
    if (followedIds[1] >= 0) this.W[userId][followedIds[1]] = 0.0;
    if (unfollowedId   >= 0) this.W[userId][unfollowedId]   = 0.0;
    setFollowerNumArray(); // ← O(N²) rescan every call
}

// AFTER: O(1) Set.add/remove + counter increment/decrement
public void updateAdjacencyMatrix(int userId, int[] followedIds, int unfollowedId) {
    if (followedIds[0] >= 0) {
        followees[userId].add(followedIds[0]);
        followers[followedIds[0]].add(userId);
        followerNumArray[followedIds[0]]++;
    }
    if (followedIds[1] >= 0) { ... } // symmetric remove
    if (unfollowedId   >= 0) { ... } // symmetric remove
}

Rationale: setFollowerNumArray() looped over all N² cells after every follow/unfollow event. With ~2.5 active agents per step each potentially triggering a follow or unfollow, this was O(N²) per step merely to keep counts in sync. Incremental updates reduce this to O(1) per event.

A5 — Removed setW() and setFollowerNumArray()

Issue type: Dead code

setW() had no callers after the refactor. setFollowerNumArray() is replaced by incremental updates in A4. Both removed.

4 — OpinionDynamics.java PERF + CORRECTNESS

C1 — Per-step W matrix materialisation removed

Location: OpinionDynamics.java:evolve() — top of step loop

Issue type: Performance

// BEFORE: inside for (int step = 1; step <= t; step++)

C2 — Post/repost distribution: O(N) scan → O(degree) iteration

Location: OpinionDynamics.java:evolve() — post and repost blocks

Issue type: Performance

// BEFORE: scan all N agents to find followers
for (Agent otherAgent : agentSet) {
    if (W[otherAgent.getId()][agentId] > 0.00)
        otherAgent.addToPostCash(post);
}

// AFTER: iterate only actual followers — O(degree ≈ 10)
for (int followerId : admin.getFollowers(agentId)) {
    agentSet[followerId].addToPostCash(post);
}

Applied identically to both post creation and repost distribution. With MAX_FOLLOW_CAPACITY = 10, each agent has at most ~10 followers, reducing per-post distribution from O(1000) to O(10).

C3 — AssertionChecker moved outside the agent loop

Location: OpinionDynamics.java:evolve()

Issue type: Correctness + Performance

// BEFORE: called N=1000 times per step (inside shuffled-agent loop)
agent.resetFeed();
ASChecker.assertionChecker(agentSet, admin, agentNum, step); // ← N redundant calls
    if (followedIds[0] >= 0) { followActionNum++; } // misindented under assertionChecker

// AFTER: called once after all agents have acted
agent.resetFeed();
if (followedIds[0] >= 0) { followActionNum++; }   // corrected indent
if (unfollowedId   >= 0) { unfollowActionNum++; } // corrected indent
}  // end agent loop
ASChecker.assertionChecker(agentSet, admin, agentNum, step); // once per step

Rationale: Step-level invariant checks belong after the full agent loop, not inside it. Calling N times per step (a) runs 999 redundant checks against partially-updated state, and (b) since assertionChecker itself calls admin.getAdjacencyMatrix() (O(N²)), this was O(N³) per step — the dominant cost of the simulation. Moving it outside reduces to O(N²) per step. Also fixes the indentation bug where followActionNum++ was visually attached to the checker call instead of the outer loop body.

C6 — writer.flush() added after simulation loop

Location: OpinionDynamics.java:evolve() — after for (int step = 1; ...)

Issue type: Correctness

writer.flush(); // flush final partial batch (e.g. step 40000 alone)

Rationale: Without this call, any rows accumulated in the Writer's StringBuilders after the last full-batch flush are silently discarded when the JVM exits. For a 40 000-step run with BATCH_SIZE=1000, step 40000 falls into a partial batch of 1 row that would never be written.

5 — Writer.java BATCH OUTPUT

D1 — Accumulate-and-flush batch architecture

Location: Writer.java:write()

Issue type: Performance / Structure

// BEFORE: open → write header + 1 row → close, per step, 3 files × 40 000 steps
public void write() {
    String filePath = folderPath + "/metrics/result_" + simulationStep + ".csv";
    try (PrintWriter pw = new PrintWriter(new FileWriter(filePath, false))) {
        pw.println(header); pw.println(row);
    }
}

// AFTER: accumulate into StringBuilder; flush every BATCH_SIZE steps
private static final int BATCH_SIZE = 1000;
private int batchStartStep = 0;
private StringBuilder metricsBuf = new StringBuilder();
private StringBuilder postsBuf   = new StringBuilder();
private StringBuilder opinionBuf = new StringBuilder();
// write() appends one row; flushBatch() is called automatically when full

Rationale: 120 000 individual file open/write/close operations per run. Batching to 1000 steps per file reduces this to 123 total writes (40 batch files + 3 per checkpoint × 8 + 3 for the partial final batch). Batch file naming convention: result_{batchStart}_{batchEnd}.csv makes the step range unambiguous.

D2 — flushBatch() private + writeBuffer() shared I/O path

Location: Writer.java — new private methods

Issue type: Structure

private void flushBatch() {
    int batchEnd = simulationStep;
    writeBuffer(metricsBuf, folderPath + "/metrics/result_"  + batchStartStep + "_" + batchEnd + ".csv");
    writeBuffer(postsBuf,   folderPath + "/posts/post_result_" + batchStartStep + "_" + batchEnd + ".csv");
    writeBuffer(opinionBuf, folderPath + "/opinion/opinion_result_" + batchStartStep + "_" + batchEnd + ".csv");
    metricsBuf.setLength(0); postsBuf.setLength(0); opinionBuf.setLength(0);
    batchStartStep = simulationStep + 1;
}
private void writeBuffer(StringBuilder buf, String filePath) {
    try (PrintWriter pw = new PrintWriter(new FileWriter(filePath, false))) { pw.print(buf); }
    catch (IOException e) { e.printStackTrace(); }
}

All three output types (metrics, posts, opinion) share one I/O path, keeping error handling in one place.

6 — Const.java CLEANUP + ANNOTATIONS

E1 — Removed unused DECREMENT_BC

Location: Const.java

Issue type: Dead code

public static final double DECREMENT_BC = 0.003;

The only reference was in a commented-out line in Agent.updateMyself(): //this.bc -= Const.DECREMENT_BC * decayFunc(this.timeStep); Both the call site and decayFunc() itself (B3) are gone. BC decay now uses only BC_DEC_RATE multiplicatively.

E2 — Annotated frozen useProb bounds (DEFERRED)

Location: Const.java:MAX_PU / MIN_PU

// DEFERRED §8.1: MAX_PU == MIN_PU == INITIAL_PU pins useProb to a constant — adaptive engagement is inert
public static final double MAX_PU = INITIAL_PU;
public static final double MIN_PU = INITIAL_PU;

Rationale: The clamp in Agent.updateMyself() reduces the multiplicative *= 1.1 / *= 0.9 updates to zero effect. The platform engagement mechanism is modelled but numerically inert. This must be flagged in any simulation report. A live fix (setting distinct bounds) requires verifying whether the research design intends fixed activity rates.

E3 — Annotated one-directional BC decay (DEFERRED)

Location: Const.java:BC_DEC_RATE

public static final double BC_DEC_RATE = 0.99;  // bc only decays; no recovery — see deferred issue §8.2

Rationale: BC decays multiplicatively on discordant post exposure and is never increased. All agents asymptotically converge to MINIMUM_BC = 0.2 regardless of initial conditions, making the long-run BC distribution parameter-independent. Adding a recovery term (bc += α on concordant exposure) would make BC a genuine dynamic variable but constitutes a model change — deferred pending research design review.

7 — Deferred items FLAG IN ALL REPORTS

These issues were identified but intentionally left unchanged. They must be flagged in any future simulation report or parameter sensitivity analysis, and require research-design-level decisions before a code fix is appropriate.
