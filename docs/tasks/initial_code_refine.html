2 — Const.java DONE

Added shannonIndex and disagreement to RESULT_LIST

Both metrics were already computed (analyzer.computeShannonWienerIndex() and what was then analyzer.computeDisagreement()) and passed to Writer via setters. However, because neither key appeared in RESULT_LIST, Writer.write()'s loop never reached their switch cases and they were silently omitted from CSV output.

Fix: Insert both keys after "opinionAvg":

- "opinionAvg", "feedPostOpinionMean_0", ...
+ "opinionAvg", "shannonIndex", "disagreement", "feedPostOpinionMean_0", ...

No other files needed changing because the Writer's appendSingleMetric() switch already handled these two cases.

Reference pattern: When adding a new metric to a similar simulation, the minimum change is (a) register in RESULT_LIST, (b) add a setter in Writer, (c) add a switch case in the metric handler. All three must be present or the metric silently disappears.

3 — AdminOptim.java ARCHITECTURE

3.1 — Replace dense double[N][N] with Set<Integer>[] adjacency lists

Before: AdminOptim held a double[][] W field (1000×1000 = 1M cells). Every follower lookup required scanning an entire row or column, i.e. O(N) per query.

After: Two HashSet<Integer>[] arrays — followees[i] (agents i follows) and followers[j] (agents that follow j). Both are built once in the constructor from the initial W in O(N²) (one-time cost), then updated incrementally.

The field private int[] followerNumArray is kept as a cached in-degree count used by the follower ranking and manipulation-target logic. It is updated synchronously in updateAdjacencyMatrix().

Declaration requires @SuppressWarnings("unchecked") on the constructor because Java cannot create generic arrays directly: 

// in constructor
@SuppressWarnings("unchecked")
this.followees = new HashSet[n];
this.followers = new HashSet[n];

3.2 — getAdjacencyMatrix() becomes on-demand only

A getAdjacencyMatrix() method is kept but is now a generator — it rebuilds double[n][n] from the adjacency sets each time it is called. This is intentional: the method is only invoked at 5000-step checkpoints for GEXF export and degree/clustering writes, so paying the O(N²) rebuild cost 8 times per 40 000-step run is acceptable. 

public double[][] getAdjacencyMatrix() {
    double[][] matrix = new double[n][n];
    for (int i = 0; i < n; i++)
        for (int j : followees[i])
            matrix[i][j] = 1.0;
    return matrix;
}

3.3 — New getFollowers() and getFollowees() accessors

public Set<Integer> getFollowers(int userId) and getFollowees(int userId) return the live set directly (no copy). Callers iterate these sets for post/repost distribution — O(degree) rather than O(N).

Mutation risk: The sets are returned by reference. Callers must not modify them during iteration. In the current codebase this is safe because distribution loops only read; updateAdjacencyMatrix() is the sole writer and is called after the per-agent inner block.

3.4 — updateAdjacencyMatrix() becomes O(1) incremental

Three conditional branches handle: (a) new follow [followedIds[0]], (b) displaced follow when capacity exceeded [followedIds[1]], (c) unfollow [unfollowedId]. Each branch does a constant-time Set.add() or Set.remove() on both followees and followers, plus an increment or decrement of followerNumArray.

Java integer autoboxing note: Set<Integer>.remove(int) autoboxes the int to Integer and calls remove(Object) — correct behaviour, no explicit cast needed.

3.5 — computeDisagreement() moved from Analysis to AdminOptim

The prior Analysis.computeDisagreement(agentSet, W) accepted the full matrix and iterated N² cells. The new AdminOptim.computeDisagreement(agentSet) iterates only directed edges in followees: 

double sum = 0.0; int edgeCount = 0;
for (int i = 0; i < n; i++)
    for (int j : followees[i]) {
        double diff = agentSet[i].getOpinion() - agentSet[j].getOpinion();
        sum += diff * diff; edgeCount++;
    }
return edgeCount > 0 ? sum / edgeCount : 0.0;

Moving it to AdminOptim is the right ownership since only Admin has access to the adjacency sets. The call in OpinionDynamics was updated from analyzer.computeDisagreement(agentSet, W) to admin.computeDisagreement(agentSet).

3.6 — Removed setW() method

setW(double[][] W) was a setter for the old matrix field. It had no callers after the refactor and was removed with no other side effects.

4 — Agent.java CLEANUP

4.1 — Removed followerNum field and its getter/setter

private int followerNum was a per-agent cached in-degree count. It was set by setFollowerNum(double[][] W) which looped over the entire column W[*][agentId] — an O(N) scan — called once per agent per step from inside the agent loop, giving O(N²) total per step.

This duplicated AdminOptim.followerNumArray[i] which is maintained incrementally. Removing the field and its setter/getter eliminates the O(N²) per-step column scan entirely.

Removed symbols:

  • private int followerNum;
  • public int getFollwerNum() (note: misspelled in original)
  • public void setFollowerNum(double[][] W)

4.2 — Removed commented-out follow() block

A large /*public int follow() { ... }*/ block — a previous version of the follow algorithm — had been retained across multiple refactor iterations inside a multi-line comment. No references existed. Removed.

4.3 — Removed decayFunc()

private double decayFunc(double time) returned Math.exp(-lambda * time) where lambda was a local variable always initialised to 0. The function therefore always returned 1.0. All callers had already been commented out in earlier refactors. Removed.

Pattern: When a decay/schedule function's parameter is zero or its callers are all commented out, remove the function rather than leaving it as documentation — it implies the feature is active when it is not.

5 — OpinionDynamics.java PERF + CORRECTNESS

5.1 — Per-step W matrix materialisation removed

double[][] W = admin.getAdjacencyMatrix(); appeared at the top of the per-step loop. This allocated a 1000×1000 double array on every step (40 000 times per run), and every post/repost distribution scanned a full column of it. This line is deleted entirely.

5.2 — Post distribution: O(N) scan → O(degree) iteration

// BEFORE: scan all N agents
for (Agent otherAgent : agentSet) {
    if (W[otherAgent.getId()][agentId] > 0.00)
        otherAgent.addToPostCash(post);
}

// AFTER: iterate only actual followers
for (int followerId : admin.getFollowers(agentId)) {
    agentSet[followerId].addToPostCash(post);
}

Applied identically to both post creation and repost distribution.

5.3 — AssertionChecker moved outside the agent loop

Bug: ASChecker.assertionChecker(agentSet, admin, agentNum, step) was called inside the per-agent loop, after each individual agent updated. This means it ran N = 1000 times per step and validated global invariants against a partially-updated state (the remaining agents in the shuffled list had not yet acted). This is both semantically incorrect (step-level invariant should be checked once after all agents have acted) and wasteful (999 redundant checks per step). 

// BEFORE: inside agent loop
agent.resetFeed();
ASChecker.assertionChecker(agentSet, admin, agentNum, step);
    if (followedIds[0] >= 0) { followActionNum++; }  // misindented

// AFTER: inside agent loop, clean
agent.resetFeed();
if (followedIds[0] >= 0) { followActionNum++; }
if (unfollowedId >= 0) { unfollowActionNum++; }
}  // end agent loop
ASChecker.assertionChecker(agentSet, admin, agentNum, step); // once per step
Indentation fix: The followActionNum++ and unfollowActionNum++ blocks had inconsistent indentation in the original (one was indented under assertionChecker, the other was not) — both are now correctly inside the agent loop at the same indent level.

5.4 — 5000-step checkpoint uses locally-scoped W5k

GEXF export, degree CSV, and clustering computation still require a dense matrix. Rather than keeping W alive for the whole step, a local variable is generated only when needed: 

if (step % 5000 == 0) {
    double[][] W5k = admin.getAdjacencyMatrix(); // local, discarded after block
    network.setAdjacencyMatrix(W5k);
    gephi.updateGraph(agentSet, network);
    gephi.exportGraph(step, folerPath);
    ...
    writer.writeDegrees(W5k, folerPath);
    writer.writeClusteringCoefficients(
        analyzer.computeClusteringCoefficients(W5k), folerPath);
}

5.5 — writer.flush() after simulation loop

Without this call, the final partial batch in Writer's StringBuilder buffers is silently discarded when the JVM exits. Added immediately after the closing brace of the for (int step = 1; step <= t; step++) loop.

5.6 — setFollowerInfo() cleaned

agent.setFollowerNum(tempAdjacencyMatrix) removed from setFollowerInfo(). Only agent.setFollowList(tempAdjacencyMatrix) is kept (this initialises the agent's own boolean follow-list which is still needed for the follow/unfollow logic in Agent.java).

5.7 — agent.setFollowerNum(W) removed from inner agent loop

This call appeared at the top of the per-agent block inside the shuffled-agent loop. It was the sole consumer of the removed field. Deleted.

6 — Writer.java BATCH OUTPUT REFACTOR

6.1 — Accumulate-and-flush batch architecture

Before: write() opened a PrintWriter, wrote header + one row, and closed the file — once per step. 40 000 steps × 3 output types = 120 000 file open/close operations per run.

After: Three StringBuilder fields (metricsBuf, postsBuf, opinionBuf) accumulate rows. On first row of each batch the header is prepended to the buffer. When simulationStep - batchStartStep + 1 >= BATCH_SIZE (1000), flushBatch() is called automatically.

New fields:

private static final int BATCH_SIZE = 1000;
private int batchStartStep = 0;
private StringBuilder metricsBuf  = new StringBuilder();
private StringBuilder postsBuf    = new StringBuilder();
private StringBuilder opinionBuf  = new StringBuilder();

6.2 — flushBatch() private method

Writes each buffer to a file, resets buffer length to 0 (via StringBuilder.setLength(0)), and advances batchStartStep. File naming convention: 

metrics/result_{batchStart}_{batchEnd}.csv
posts/post_result_{batchStart}_{batchEnd}.csv
opinion/opinion_result_{batchStart}_{batchEnd}.csv

The actual write is delegated to private void writeBuffer(StringBuilder buf, String filePath) which wraps a PrintWriter(new FileWriter(path, false)) in a try-with-resources.

6.3 — flush() public method

Handles end-of-simulation partial batch (e.g. steps 39 000–39 999 would flush naturally; steps 40 000 alone would not unless flush() is called). Guard: only calls flushBatch() if metricsBuf.length() > 0.

Called from OpinionDynamics.evolve() after the main loop.

6.4 — Removed unused feedVar scalar field

private double feedVar was initialised to -1 in the constructor and had a setFeedVar(double value) setter, but the setter was never called from OpinionDynamics. The "feedVar" key was absent from RESULT_LIST, so even the switch case for it in handleSingleMetric was dead code. All three removed together.

Note: feedVarArray (per-bin variance) is different and was kept — it is actively set and written via the "feedPostOpinionVar_N" keys.

6.5 — handleSingleMetric renamed to appendSingleMetric

The old name suggested the method handled (i.e. wrote) a metric. The new name reflects what it actually does: appends a formatted value to a StringBuilder passed as an argument. Signature unchanged; purely a naming correction.

6.6 — Switch expressions preserved (Java 14+ arrow syntax)

The existing switch (key) { case "x" -> { sb.append(...); return true; } } pattern was kept as-is. No changes to the switch body other than removal of the "feedVar" case.

7 — Notebooks BATCH CSV READER

7.1 — General pattern: step_file_map → concat + iterrows

All affected loaders previously used a two-phase approach:

  1. Regex-extract the step number from a single-step filename (post_result_123.csv), build a step_file_map dict.
  2. Iterate sorted steps, open one file per step, take df.iloc[0] (the single data row).

New pattern:

  1. Read all post_result_*.csv files (glob matches both legacy single-step and new batch filenames).
  2. Concat all DataFrames, sort by "step" column.
  3. Apply MAX_INDEX filter on the unified DataFrame (simpler than filtering filenames by step range).
  4. Iterate rows with for _, row in all_df.iterrows().

Backward compatibility: the glob post_result_*.csv matches both post_result_123.csv (old) and post_result_0_999.csv (new), so existing results directories continue to work without renaming.

8 — Deferred items FLAG IN ALL FUTURE REPORTS

These issues were identified and documented but intentionally left unchanged. They must be flagged in any future simulation report or parameter sensitivity analysis.

8.1 — postProb and useProb dynamics are numerically inert

In Const.java:

public static final double INITIAL_PU = 0.0025;
public static final double MAX_PU = INITIAL_PU;   // = 0.0025
public static final double MIN_PU = INITIAL_PU;   // = 0.0025

Because MAX_PU == MIN_PU == INITIAL_PU, the clamp in Agent.updateMyself(): 

this.useProb = Math.max(Const.MIN_PU, Math.min(this.useProb, Const.MAX_PU));

…pins useProb to exactly 0.0025 regardless of the multiplicative updates (*= 1.1 / *= 0.9) that precede it. The adaptive engagement mechanism is mathematically modelled but has zero effect on simulation outcomes.

postProb has slightly more room (MIN_PP=0.05, MAX_PP=1.0), but POST_COST = 0.0 removes the cost penalty, leaving only the comfort-rate multiplier active.

Impact for reports: Any claim that "platform engagement adapts to echo-chamber exposure" is numerically unsupported by the current parameter settings.

8.2 — Comfort threshold is the static floor MINIMUM_BC, not dynamic bc

In Agent.updateMyself():

if (Math.abs(post.getPostOpinion() - this.opinion) < Const.MINIMUM_BC) {

The intent is to count posts within the agent's current bounded-confidence window. However, this uses the global constant MINIMUM_BC = 0.2 instead of this.bc, which can decay via bc *= BC_DEC_RATE per discomfort event.

Additionally, the bc clamp on line 313 reads: 

this.bc = Math.max(this.bc, Math.min(Const.MINIMUM_BC, 1.0));

Since Math.min(0.2, 1.0) = 0.2 always, this is equivalent to this.bc = Math.max(this.bc, 0.2) — which enforces the floor but never raises bc (there is no ceiling clamp). The asymmetry means bc only decreases (or stays at floor).

Net effect: Comfort rate is computed against a fixed 0.2-width window while the opinion update uses a potentially narrower window. The comfort signal systematically over-counts comfortable posts for agents with bc < 0.2 (which should be impossible given the floor, but structurally the two quantities are decoupled).

9 — Reusable patterns for similar ABM projects

P1 — Graph adjacency list vs dense matrix

  • Use Set<Integer>[] (HashSet) for runtime follow/unfollow, post distribution, and edge-based metric computation.
  • Keep getAdjacencyMatrix() as an on-demand generator for export-only operations.
  • Cache in-degree counts separately if ranking is needed (O(1) read vs O(degree) set-size call).
  • Incremental updates: Set.add()/Set.remove() on both followees and followers in one method — never update one without the other.

P2 — Post-loop assertion checking

  • Step-level invariant checks belong after the full agent loop, not inside it.
  • Inside the loop = O(N) redundant checks against partially-updated state.
  • Outside the loop = single check against the fully-committed step state.

P3 — Batch CSV output with StringBuilder

  • Accumulate rows in StringBuilder fields; flush every N steps.
  • Always expose a public flush() for end-of-simulation partial batches.
  • Headers are written once per batch (check buf.length() == 0).
  • File naming: result_{batchStart}_{batchEnd}.csv makes both the range and ordering obvious without a manifest file.
  • Keep writeBuffer(StringBuilder, String) as a private utility so all three output types (metrics, posts, opinion) share one I/O path.

P4 — Notebook batch CSV reader

  • Glob pattern result_*.csv matches both old single-step names and new batch names — backward compatible.
  • Concat all files first, then filter by step — do not try to filter files by name when files cover ranges.
  • Replace per-file pd.read_csv + iloc[0] with concat + iterrows() when moving to multi-row files.

P5 — Metric registration checklist

Three locations must all be updated when adding a new scalar metric:

  1. Add key to RESULT_LIST in Const.java
  2. Add setXxx() setter in Writer
  3. Add case "xxx" -> { ... return true; } in appendSingleMetric()

Missing any one of these causes the metric to be silently absent from CSV output.

P6 — Dead code indicators

  • A helper whose lambda / exponent parameter is always 0 → always returns 1 → dead.
  • A setter that is never called from outside the class → field is effectively read-only at init value.
  • A field duplicated in another class with a more efficient maintenance strategy → remove the duplicate.
  • A commented-out block retained across multiple refactor iterations → remove it; history is in git.
