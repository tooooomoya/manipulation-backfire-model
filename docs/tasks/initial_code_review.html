Overview. The codebase implements a bounded-confidence, co-evolving-network opinion dynamics model in the Friedkin-Johnsen (FJ) family. The simulation machinery is structurally sound, but several mechanisms that are central to the stated research goals — platform churn, engagement feedback, and the asymmetric stubbornness contribution — are silently disabled or broken in the current code. Two key output metrics (Shannon diversity index and disagreement) are computed but never written to disk. Performance is dominated by O(N³·T) assertion checking. These issues must be resolved before results can be trusted.

1. Model Understanding

What is implemented?

The simulation models N=1000 agents on a directed social network, each holding a continuous opinion x ∈ [−1, 1] and an intrinsic (anchoring) opinion x₀ ∈ [−1, 1]. At each discrete time step (up to T=40,000), a subset of agents becomes active (probability u=0.0025 per agent per step — ≈2.5 agents active on average), posts content, reads a feed, and updates their opinion, posting probability, and bounded confidence threshold.

Core opinion update rule

// Agent.java:313
opinion = μ · x₀ + (1 − μ) · mean(feed_opinions)

This is the Friedkin-Johnsen (FJ) update, where μ (stubbornness) interpolates between the agent's intrinsic opinion and the mean of posts in their feed. The bounded confidence (BC) threshold bc determines which posts are tolerated; posts outside bc trigger a decay bc *= BC_DEC_RATE (no mechanism to increase it). Feed content is filtered through PostCash, a bounded FIFO queue of posts received from followees.

updatePostProb() computes an increment from received likes but the line applying it is commented out. The method only clamps and resets the counter.

Impact: The positive engagement feedback loop (likes → increased posting) does not exist in the model despite being described in comments. Any hypothesis about viral content amplification through likes is untestable with this code.

OpinionDynamics.java:203
Performance · High

Assertion checker called per-agent, not per-step → O(N³·T)

ASChecker.assertionChecker() is invoked inside the per-agent loop. The checker iterates the full N×N adjacency matrix, making it O(N²) per call, O(N³) per step, and O(N³·T) overall. 

for (Agent agent : shuffledAgents) {
    // ... agent update logic ...
    ASChecker.assertionChecker(agentSet, admin, agentNum, step);  // ← called N times per step
}

Impact: For N=1000, T=40,000 → approximately 40 billion matrix element accesses just for consistency checking. This is orders of magnitude above what the opinion dynamics themselves require. In practice it likely dominates wall time. Should be moved outside the agent loop or disabled in production runs.

Agent.java:313 + OpinionDynamics.java:140–211
Model · High

Default TARGET_DIRECTION=0 disables asymmetric stubbornness

The branch is named asymmetric-stubbornness, but the feature only activates when TARGET_DIRECTION ≠ 0. With the default value of 0, x = opinion * 0 = 0 for all agents, and all agents receive stubbornness = 0.6 (the else branch). 

private double calculateStubbornness(double opinion) {
    double x = opinion * Const.TARGET_DIRECTION;  // = 0 when TARGET_DIRECTION=0
    if (x <= -0.6) value = 0.8;
    else if (x >= 0.6) value = 1.0;
    else value = 0.6;  // ← always reached when TARGET_DIRECTION=0
}

Impact: Running java dynamics.OpinionDynamics without a second argument (target) gives a symmetric stubbornness model, not the asymmetric one. The run script does pass ±1.0, so production runs are correct, but this is a fragility. The code silently falls back to a different model.

Agent.java:273
Model · High

Comfort uses hardcoded threshold MINIMUM_BC, not dynamic bc

The comfort post count — which drives postProb and useProb — uses the fixed constant MINIMUM_BC = 0.2 rather than the agent's current bounded confidence value. 

if (Math.abs(post.getPostOpinion() - opinion) < Const.MINIMUM_BC) {  // 0.2 — always fixed
    comfortPostNum++;
}

Impact: As bc decays toward 0.2, agents become less tolerant, but their comfort threshold stays at 0.2. The two variables are supposed to be linked conceptually but are implemented as independent quantities. This decoupling distorts the relationship between echo-chamber formation (bc decay) and engagement (comfort-based postProb update).

3. Design Evaluation

Design

Dense N×N adjacency matrix for a sparse social graph

With MAX_FOLLOW_CAPACITY = 10, each agent follows ≤10 others. The expected edge density is ≤1%. Yet the adjacency matrix is a dense double[1000][1000] = 8 MB per copy. More critically, getAdjacencyMatrix() deep-copies the full matrix and is called repeatedly per step (once for the base W, once per agent for setFollowerNum, once for AssertionCheck, once per GEXF export). Each copy is ~8 MB allocated.

An adjacency-list representation (e.g., Set<Integer>[] followees) would reduce memory by ~100× and graph traversal from O(N) to O(degree).

Design

Dual state representation for follow relationships

Follow state is stored in two places: Agent.followList[] (per-agent boolean array) and AdminOptim.W[][] (global adjacency matrix). Updates must propagate to both. AssertionCheck exists specifically to detect divergence between them — a design smell indicating that maintaining consistency is already known to be error-prone. A single source of truth would eliminate this class of bug.

Design

Configuration in code, not in YAML

config_template.yaml exists and documents parameters (N, μ, ε_s, topology, etc.), but it is never read by the Java simulation. All parameters are hardcoded in Const.java. Running different parameter sweeps requires recompiling. The YAML file and Const.java are already diverging (e.g., YAML has mu: 0.5, code has INITIAL_STUBBORNNESS: 0.8; YAML has opinion_std: 0.6, code has INITIAL_OPINION_STD: 0.4).

Design

120,000 file writes per simulation run — no buffering at the CSV level

Writer.write() opens, writes one row, and closes three separate CSV files per step. Over 40,000 steps, that is 120,000 individual file open/write/close operations. No step-level output buffering is used. Aggregating output into larger batches (e.g., write every 100 steps, or use a single append-mode file with step as a column) would dramatically reduce I/O overhead and simplify the Python analysis (which currently reads 40,000+ individual files).

4. Research Risks

Risk

Gauss-Seidel vs. Jacobi updating — undocumented methodological choice

Agents are processed in a random shuffled order each step. Because opinion updates are applied immediately (agent.updateMyself() modifies agent.opinion in place), later agents in the step's shuffle read the already-updated opinions of earlier agents. This is a Gauss-Seidel (asynchronous) updating scheme, not the synchronous Jacobi scheme typical in bounded-confidence literature (Deffuant, HK).

Gauss-Seidel updating can produce qualitatively different equilibria and convergence rates. This choice should be explicitly documented and motivated — or changed to a snapshot-based approach where all opinions are read from a pre-step snapshot.

Risk

Hub seeding creates opinion-topology inconsistency

The CNNR network is built using opinion similarity (chooseNodeBySimilarity): new nodes preferentially connect to existing nodes with similar opinions. After the network is fully constructed, the top-k hubs by follower count are then assigned extreme opinions (±1.0) and their stubbornness is recomputed. The network topology was shaped by initial random opinions, not the seeded extreme opinions. Hubs may therefore be structurally embedded in communities with incompatible opinions, creating an artificial starting tension that is an artifact of the initialization procedure rather than an emergent property.

Risk

Extremely low activity rate suppresses dynamics

With INITIAL_PU = 0.0025 (frozen), ~2.5 agents are active per step on average. At MAX_FOLLOW_CAPACITY = 10, each active agent reads posts from ≤10 followees. The expected number of posts read per step (across all agents) is tiny. With INITIAL_PP = 0.1 and ~2.5 active agents, only ~0.25 new posts are generated per step on average. Over 40,000 steps, that is ~10,000 total posts for 1,000 agents — 10 posts per agent over the entire simulation. Most agents will go many steps without any feed content at all, making the social influence mechanism irrelevant for much of the run.

Risk

Python analysis reads only 12.5% of simulation output by default

opinion_variance_analysis.py sets max_step = 5000, discarding steps 5001–40,000. If the model needs ~10,000 steps to reach steady state (plausible given the low activity rate), the standard analysis output captures only transient behavior. Results presented from this analysis may not reflect the model's equilibrium properties.

Risk

Bounded confidence is monotonically decreasing — no recovery mechanism

BC decays on every discordant post (bc *= 0.98) and is never increased. In the long run, all agents converge toward bc = MINIMUM_BC = 0.2, regardless of initial conditions or network structure. This means the model's behavior at long time scales is dominated by this floor rather than by the social influence dynamics. The asymptotic state of the BC distribution is fully determined by the floor, not by the research parameters. Any claims about steady-state BC distributions require careful qualification.

# Compute n_c from opinion_result CSVs, or add highComfortRateNum as a count proxy

Structural improvements (design)

Design

7. Replace dense adjacency matrix with adjacency lists

Store follower/followee relationships as Set<Integer> per agent. This eliminates the O(N²) scans in post distribution, follower counting, and assertion checking. Update AdminOptim to delegate state ownership to agents rather than maintaining a separate matrix.

Design

8. Consolidate to a single source of truth for follow state

Remove either Agent.followList or AdminOptim.W as authoritative state. Keep one and derive the other on-demand. This eliminates the need for AssertionCheck's consistency verification entirely.

Design

9. Wire the YAML config to Java simulation

Add a YAML parser (e.g., SnakeYAML, already likely in the lib directory for Gephi) and read parameters from config_template.yaml at startup. This decouples experiment configuration from compilation and prevents the YAML/Const.java drift that is already visible (mu, opinion_std disagree between files).

Design

10. Batch CSV output — one file per simulation, not per step

Write all metrics rows to a single append-mode CSV per metric type, with step as the first column. This reduces 120,000 file I/O operations to 3 and simplifies the Python analysis from glob-sorting 40,000 files to reading one.

// Write once at end, or append-mode each step
try (PrintWriter pw = new PrintWriter(new FileWriter(filePath, true))) {
    pw.println(row);  // append, header written once at init
}

Model-level recommendations

Model

11. Add a BC recovery mechanism

Consider a slow recovery: bc = min(bc + α, 1.0) on concordant post exposure. Without it, BC converges to 0.2 for all agents regardless of conditions, making the long-run equilibrium parameter-independent. A recovery term makes BC a genuine dynamic variable and allows studying its steady-state distribution as a function of network topology and stubbornness.

Model

12. Document and justify the Gauss-Seidel update order

Either switch to snapshot-based (Jacobi) updating — take a copy of all opinions before the step begins and use those for feed influence — or explicitly state and motivate the asynchronous scheme. For comparison, run both variants with identical seeds and report whether conclusions differ.

// Jacobi-style: copy opinions before the step
double[] snapshotOpinions = Arrays.stream(agentSet)
    .mapToDouble(Agent::getOpinion).toArray();
// Use snapshotOpinions[i] for influence computation, not agent.getOpinion()
Model

13. Separate hub seeding from network construction

Assign extreme opinions to hubs before building the CNNR network, or use an opinion-agnostic hub-selection criterion (e.g., random among top-k by degree in a pre-built topology). The current design creates a structural inconsistency where the network was built to reflect one opinion landscape but must operate in another.

Model

14. Increase Python analysis horizon to the full simulation

Set max_step = Const.MAX_SIMULATION_STEP (or pass it as an argument) in all Python scripts. Use adaptive smoothing (larger window at scale) to handle the full 40,000-step series without overwhelming the plots.
